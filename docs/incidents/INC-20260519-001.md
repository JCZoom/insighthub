# Incident INC-20260519-001 — Production auth bypass (NEXT_PUBLIC_DEV_MODE baked into build)

**Severity:** S2 (HIGH) — confirmed auth bypass with no observed exploitation
**Detection date/time:** 2026-05-20 ~02:23 UTC (2026-05-19 ~22:23 ET)
**Declaration date/time:** 2026-05-20 02:25 UTC (implicit — IC moved straight to containment)
**Containment date/time:** 2026-05-20 02:25:46 UTC (`sudo systemctl stop insighthub`)
**Recovery date/time:** _PENDING_ (code split + redeploy + smoke tests in progress at time of writing)
**Retrospective date:** 2026-05-20 (this document — final pass after recovery)

## Summary

The production deployment of InsightHub at `https://dashboards.jeffcoy.net` was running with authentication completely bypassed. Every protected route (`/dashboards`, `/admin`, `/api/data/query`, `/api/freshworks/*`, `/api/chat`, etc.) was accessible to anyone on the public internet without a session, and every server-side action was attributed to the synthetic `dev-admin-user` in audit logs.

The bypass was controlled by `process.env.NEXT_PUBLIC_DEV_MODE === 'true'` in `middleware.ts` and three other server modules. The `NEXT_PUBLIC_` prefix causes Next.js to inline the value at build time, so the bypass was frozen into every production bundle that was ever built — `.env.local` had no live influence over it. The CI workflow's top-level `env:` block set `NEXT_PUBLIC_DEV_MODE: "true"`, which the deploy job inherited when running its own `npm run build` on the EC2 host, so every CI deploy re-baked the bypass.

Forensic review of HTTP access logs for the only data-relevant window (2026-05-19, the day Freshworks live data landed on prod) found zero unauthorized access. The only two IPs that received 200-OK responses on InsightHub paths were Jeff Coy's home Verizon IP and his Tailscale exit-node IP — both legitimate IC activity. ~70 other source IPs received only 404s on generic scanner probes (`/wp-login.php`, `/.env`, etc.). No live Freshworks data was exfiltrated.

## Impact

- **Systems affected:** Production EC2 host `autoqa`, `https://dashboards.jeffcoy.net` web tier and all API routes.
- **Data affected (classification):** Theoretically all data behind the web tier — USZoom Restricted (sample dashboards, glossary, audit log) and Customer Confidential (live Freshworks pipeline values, deal counts, ticket counts as of 2026-05-19). Actually accessed by an unauthorized party: **none observed** in HTTP logs.
- **Users affected (count + role):** Zero unauthorized users observed. Audit log shows all `dev-admin-user` rows trace back to legitimate IC sessions during the exposure window.
- **Regulatory notification required:** No. Per Policy 3719 §6 and GDPR Art. 33, notification is required for breaches of personal data with risk to data subjects. Forensics show no PII was disclosed to an unauthorized party.
- **Customer notification required:** No. No customer data was disclosed.

The "no observed exploitation" conclusion rests on `/var/log/nginx/access.log` retention being sufficient to cover the window, which it is. We do not have application-level audit rows for unauthenticated requests (middleware skipped audit logging), so the nginx access log is the canonical evidence.

## Root cause

### Technical (Swiss cheese — every hole had to align)

1. **Wrong env-var prefix for a security flag.** `NEXT_PUBLIC_DEV_MODE` controlled authentication bypass in `middleware.ts`, `src/lib/auth/config.ts` (two places), and `src/lib/auth/session.ts` (two places). The `NEXT_PUBLIC_` prefix is documented (Next.js) to mean "inline this value into the JS bundle at build time so it's visible in the browser." It is the wrong choice for a runtime security control: once baked, you cannot toggle it by editing `.env.local`.

2. **CI workflow baked the bypass into every deploy.** `.github/workflows/ci.yml:18` set `NEXT_PUBLIC_DEV_MODE: "true"` at workflow scope. The deploy job at line 254 ran `npm run build` on the EC2 host and inherited that env, so the standalone bundle that landed at `/opt/insighthub/.next/standalone/server.js` had `'true' === 'true'` literally compiled into the middleware's auth gate.

3. **OAuth credentials were never wired.** `/opt/insighthub/.env.local` had `GOOGLE_CLIENT_ID=""` and `GOOGLE_CLIENT_SECRET=""`. Even if a future operator wanted to flip dev-mode off, they would have been unable to — sign-in would 500 with no Google provider configured. This created a perverse incentive to keep dev-mode on.

4. **`NEXTAUTH_URL` pointed at `http://localhost:3000` on prod.** Same root cause — `.env.local` was never properly initialized for production. Would have broken OAuth callbacks if (1)–(3) were ever fixed in isolation.

5. **`assertEnv()` warned but did not fail.** `src/lib/env.ts:292-303` (pre-fix) logged `⚠️  NEXT_PUBLIC_DEV_MODE=true in production` but did not throw. The warning appeared on every service start since the prod machine was first stood up. Nobody was paging on stderr warnings.

6. **No external attack-surface monitor.** There is no automated check that probes `https://dashboards.jeffcoy.net/dashboards` from outside the host and asserts it returns a 307 redirect to the login page (or 401/403). Such a check would have flagged this within minutes of the first deploy.

### Organizational

- **The system was launched in "demo mode" and never re-architected for real users.** `CONTRIBUTING.md:134` (pre-fix) explicitly documented the demo posture: "the demo deployment uses `NEXT_PUBLIC_DEV_MODE=true`; for a real multi-user production deployment, set it to `false`." That note assumed an operator would manually re-configure things before going public. Two things happened that the note didn't anticipate: (a) the site was indexed and reachable from the public internet for an extended period without the cutover, (b) live customer data (Freshworks pipeline values) was hooked up while the demo posture was still in place.
- **Single operator (Jeff Coy) holds all roles (IC, technical responder, security manager). No four-eyes review of the prod env or the CI workflow.** Headcount-bound. Mitigations are technical (assertions, monitors) until additional headcount.

## What went well

- **Detection by the operator within a window that contained zero exploitation.** Forensics confirmed no unauthorized 200-OKs. We got lucky; we got lucky early; we acted fast.
- **Containment was instant and trivially reversible.** `sudo systemctl stop insighthub` brought the public surface to 502 in under a second. nginx still served 502 cleanly. No traffic could probe the app while we triaged.
- **Forensic chain held.** nginx access logs covered the entire window. IPs cleanly partitioned into "legitimate IC" (2) and "scanner 404 noise" (~70).
- **Pre-staged remediation script with backup + redacted diff + double-confirm.** `scripts/apply-oauth-fix.sh` was written before any prod file was touched. It produced a deterministic, reviewable patch. If we'd hit the build-baked issue earlier, the rollback would also have been one command (`cp .env.local.bak-pre-oauth-fix-* .env.local`).
- **The build-baked issue was caught in a 1m47s exposure window, not in a multi-hour "why is auth still broken" debugging session.** systemd's boot log printed the smoking-gun warning on the very first restart, and we acted on it immediately.

## What didn't go well

- **Pair-programmer Cascade (this session) should have spotted the `NEXT_PUBLIC_` build-bake issue _before_ restarting the service.** The pattern is documented in Next.js docs and was readable from `middleware.ts` directly. I instead trusted that flipping `.env.local` would work. That cost us the 1m47s second exposure window. Lesson: when remediating security flags, read the code that consumes them first, especially in Next.js where the public/private env split has invisible consequences.
- **Five separate security-significant misconfigurations had to coincide for the bypass to be exploitable.** Defense in depth helped (in that there were so many things broken that anyone who tried to fix one in isolation would have found another), but it also masked the severity because operators kept thinking "I just have to fix the one config." We needed an audit-style walkthrough of the full `auth chain in production`, which this incident produced retroactively.
- **No alarm fired.** `assertEnv()` warning went to stderr → journald → never paged. Nobody read journalctl during the long quiet period when the bypass was the steady state.
- **CONTRIBUTING.md's note about flipping dev mode for production was easy to miss.** It described the cutover as a one-liner ("set it to `false`"), under-selling the fact that the cutover also requires OAuth wiring, `NEXTAUTH_URL` correction, and (post-this-incident) a code refactor.

## Lessons learned / policy updates

- **Security flags must be server-only env vars.** Codify in `CONTRIBUTING.md` and `AGENTS.md`: any flag that gates authentication, authorization, audit, or rate-limit decisions must NOT have the `NEXT_PUBLIC_` prefix. If a client component needs to "know" about such a flag, the value must come from an authenticated server endpoint, not `process.env`.
- **CI workflow scope env must default to production-safe.** Future authors of CI workflow env-var lines should assume the deploy job will inherit the workflow scope and bake those values into the production bundle. Override "dangerous" defaults at the step level inside the e2e job only.
- **`assertEnv()` should fail-fast for security-critical misconfigurations** (DEV_MODE=true in NODE_ENV=production). A warning that nobody pages on is a warning nobody reads.
- **Add to retrospective checklist:** when remediating a security flag, read the code that consumes the flag _before_ making the env change. Next.js's `NEXT_PUBLIC_` build-baking is the canonical instance, but the principle generalizes.
- **Single-operator IC posture is fragile.** Continue prioritizing automated controls (assertions, external monitors, deploy-time tests) over process controls (review checklists) until additional headcount is available.
