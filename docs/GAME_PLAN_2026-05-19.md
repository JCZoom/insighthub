This is both the demo and the proof we can build features without creating new gaps. Every step has a compliance angle — flagged as **[C]** below.

### 3.2 Connector code (2.5 hr)
Build in `src/lib/integrations/freshworks/` (new folder, mirrors `src/lib/snowflake/` structure intentionally so the audit story is "we apply the same architecture to every data source"):

- `config.ts` — reads `FRESHWORKS_API_KEY`, `FRESHWORKS_DOMAIN` from env via `src/lib/env.ts` validator (add them there with required:true). **[C]** Never logged, never returned by API routes.
- `client.ts` — typed Freshworks API client. `getContacts(filter, limit)`, `getDeals(filter, limit)`, `getAccounts(filter, limit)`. Uses native `fetch`, no extra dep. Rate-limited via existing `SlidingWindowRateLimiter` from `src/lib/rate-limiter.ts` — Freshsales free tier is 100 calls/min, set our ceiling at 60. **[C]**
- `cache.ts` — Redis-backed cache (existing `src/lib/redis/`), TTL 5 min for reads. Cache key prefixed `fw:` so we can mass-invalidate. **[C]** Classification tag stored alongside cached value.
- `audit.ts` — every read logs `integration.freshworks.read` with `{ resource, count, userId }`. No PII in the log. **[C]**
- `redact.ts` — for users below a role threshold, mask email/phone to `j***@example.com` and `+1***-***-1234` style. Reuses pattern from `src/lib/snowflake/data-security.ts`. **[C]**

### 2026-05-19 09:16 ET — §3 Freshworks plan amendments
Four upgrades applied to §3 (Freshworks connector):

1. **§3.2 cache.ts — TTL reduced from 5 min to 60 seconds.** PII + customer chat content should not sit in Redis for 5 minutes. 60s is the floor that still gives meaningful rate-limit relief without creating a stale-data exposure window.
2. **§3.2 redact.ts + §3.4 — field-level masking is ON BY DEFAULT for VIEWER role.** Previous plan said "below a role threshold." New rule: VIEWER role sees masked email/phone/chat-content always. POWER_USER and ADMIN see unmasked. Unmask-for-VIEWER on a specific widget requires an explicit admin override that emits a `widget.freshworks.unmask_override` audit event.
3. **§3.2 config.ts — token isolation.** `FRESHWORKS_API_KEY` lives in a **separate** `.env.freshworks` file on the production EC2 host only, loaded by the Next.js server at boot. **Never** in `.env.local`, **never** copied via the scp pattern G-36 flags, **never** in version control. Add `.env.freshworks` to `.gitignore` and document the SSH-and-edit-in-place procedure in `docs/FRESHWORKS_OPERATOR_RUNBOOK.md`. This is a free partial mitigation of G-36 for the most sensitive credential we have.
4. **§3.6 + §4.3 demo-purge.** Tomorrow's demo MUST include a live execution of `purgeFreshworksCache()` (and matching DB cleanup if any persistent rows exist) followed by re-fetch from Freshworks. This proves the retention lever works end-to-end. Add a dedicated section to `docs/DEMO_2026-05-20_TALKING_POINTS.md` for this — it's the strongest single moment in the demo.

These upgrades add ~30 min to the §3.2 and §3.4 timeboxes. New afternoon end-time: 6:30 PM instead of 6:00 PM. Acceptable.

### 2026-05-19 ~10:14 AM ET — Smoke test: 1 bug, 3 API-shape issues found
Live diagnostics + suite-page test revealed:
1. **FRESHSALES_DOMAIN double-https** — env value contained `https://` and my baseUrl concat produced `https://https://...`. Fixed via `normalizeDomain()` defensive scheme-stripping in `shared/domain.ts`.
2. **Freshsales 403 on /deals** — Freshsales doesn't have a `GET /crm/sales/api/deals` listing endpoint at all. Lists require `/view/{view_id}` where `view_id` is tenant-specific. Fixed via `discoverViewId()` that fetches `/filters` once per process and caches the default view ID.
3. **Freshcaller status field name varies** — API returns the call's status under one of: `status`, `call_status`, `disposition`, `call_state`, or nested `call_attributes.status`. Fixed via `freshcallerCallStatus()` resolver that tries all variants.
4. **Freshchat doesn't expose GET /v2/conversations** — only `POST /v2/conversations/search`. Fixed by switching to POST search with body. Graceful degrade to empty array if search is plan-gated.

All 4 fixes shipped in `fix(freshworks): API endpoint corrections from smoke test`. Build + typecheck clean.
